# 一、通用版本号命名标准

软件版本号通常采用`X.Y.Z`的三段式格式，其中：
+ ●&emsp;**X（主版本号）**：重大更新或不兼容API修改时递增，如Vue 2→3的升级
+ ●&emsp;**‌Y（次版本号）**‌：向下兼容的功能新增时递增，如Vue 3.2→3.3
+ ●&emsp;**‌Z（修订号）**：问题修正或小优化时递增，如Vue 3.3.4→3.3.5

**特殊扩展标识：**
+ ●&emsp;**‌先行版本**：通过连字符追加（如`1.0.0-alpha`、`2.1.0-rc.1`）表示测试阶段
+ ●&emsp;**构建元数据**：使用加号追加（如`1.0.0+20240819`）记录编译信息

# 前端项目实践规范

## 1. 版本号来源

+ ●&emsp;**‌package.json**‌：推荐使用`version`字段作为基准版本

``` JSON
// package.json
{
  "version": "1.0.0-alpha.1"  // 符合SemVer规范
}
```

+ ●&emsp;**时间戳**：适用于需要唯一标识的场景（如`build.202408191243`）
+ ●&emsp;**CI/CD变量**：集成构建系统的环境变量（如`${BUILD_NUMBER}`）

## 2. 打包文件命名组合

推荐采用多维度标识组合：
``` text
[name].[contenthash].[version].[ext]
示例： main.3e4f8d2.1.0.0.js
```
其中：
+ ●&emsp;`contenthash`：基于文件内容的哈希值（解决缓存问题）
+ ●&emsp;`version`：项目版本号（便于追踪）
+ ●&emsp;`ext`：文件扩展名（js/css等）

# 三、Vue3项目配置方案

## 1. Vite项目配置
``` javascript
// vite.config.js
export default defineConfig({
  build: {
    rollupOptions: {
      output: {
        entryFileNames: `[name].${env.VERSION}.[hash].js`,
        chunkFileNames: `[name].${env.VERSION}.[hash].js`,
        assetFileNames: `[name].${env.VERSION}.[hash].[ext]`
      }
    }
  }
})
```

## 2. Webpack项目配置
``` javascript
// vue.config.js
module.exports = {
  configureWebpack: {
    output: {
      filename: `js/[name].[chunkhash].${process.env.VERSION}.js`,
      chunkFilename: `js/[name].[chunkhash].${process.env.VERSION}.js`
    }
  }
}
```

# 四、版本管理最佳实践

+ **1.开发阶段**：使用`0.Y.Z`格式，表示API不稳定
+ **‌2.预发布阶段‌**：添加`-alpha/-beta`后缀（如`1.0.0-beta.1`）
+ **‌3.生产环境‌**：严格遵循`X.Y.Z`格式，避免使用先行版本
+ **‌4.版本升级‌**：
+ &emsp;●&emsp;重大变更 → 递增主版本（`1.9.0` → `2.0.0`）
+ &emsp;●&emsp;功能新增 → 递增次版本（`1.2.3` → `1.3.0`）
+ &emsp;●&emsp;Bug修复 → 递增修订号（`1.2.3` → `1.2.4`）

# 五、特殊场景处理

## ‌1.多环境打包‌：
``` text
app-1.0.0-prod.js    // 生产环境
app-1.0.0-stage.js   // 预发布环境
app-1.0.0-dev.js     // 开发环境
```

## 2.微前端应用‌：
添加子应用标识前缀
``` text
auth-app-1.2.0.js
admin-app-2.1.0.js
```

通过合理配置版本号命名规则，可以有效解决浏览器缓存问题，同时建立清晰的版本追溯体系。建议结合项目的CI/CD流程自动化生成版本标识
